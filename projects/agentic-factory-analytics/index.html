Agentic Factory Analytics

EDAS (Event Driven Analysis System) — event-driven multi-agent analysis on factory PostgreSQL. Users describe what they want; Manager, Planner, and Executor agents coordinate through a shared Task Registry. Part 2 of the IIoT Industrial Edge Data Platform.

CompanySomic Ishikawa
RoleR&D Engineer

TypeAI / Analytics

Overview

Modern factories have IoT teams that maintain PostgreSQL databases full of machine and sensor data — but schema, table relationships, and column formats are a black box to everyone else. Custom graphs and analysis usually require waiting on specialists who know the data layout.

EDAS bridges that gap: users describe analysis goals in natural language; agents resolve line, time range, and aims against the IoT catalog, then plan and execute queries. The name reflects the event-driven architecture — how agents coordinate — not the data format itself.

This is Part 2 of the factory data initiative at Somic Ishikawa. It builds on the IIoT platform, where TRITON edge ingestion and n8n + FastAPI processing persist granular shop-floor data in PostgreSQL — replacing license- and register-limited legacy IoT.

Instead of a rigid pipeline where every step waits on the previous one, EDAS uses a shared Task Registry: each agent drops and picks work independently. That enables parallel execution, targeted scaling, and resume-from-failure instead of restarting from scratch.

Meet the Agents

Three specialized agents — conversational intake, query planning, and execution with feedback loops.

Manager Agent

The Manager sits between what the user wants and what the IoT catalog knows. Every message routes through conversation lanes: direct planning when the user is clear, exploration when they need options, or advisory when they ask questions first. It validates machine/line names against the catalog, parses relative time phrases against a reference timestamp, and fills slots — line, time, and aim — until the user confirms with “go”.

On confirmation, the Manager writes the structured task to task_registry (the only database write in a session) and builds the Planner payload. Behind each reply, a 27-node LangGraph flow runs — only 6 LLM calls; the rest is deterministic routing and validation.

Planner Agent

The Planner is the strategist. It picks confirmed tasks from the Task Registry and reads each research topic the Manager structured. For every aim, it builds a detailed plan: which tables to query, how to join them, what filters and time bounds apply, and the step-by-step retrieval sequence the Executor should run. Completed plans are dropped back into the registry — the Planner never talks to the Executor directly; the registry is the only handoff point.

Executor Agent

The Executor is the doer. It picks plans from the registry and runs them against factory PostgreSQL. When something fails, it routes intelligently: data or table-related errors go back to the Planner for a revised plan; format or code-related errors are fixed locally and retried. Successful results are dropped to the registry for the Manager to return to the user.

Challenges Tackled

Design constraints that motivated event-driven agents over a traditional pipeline.

Pipeline bottlenecks vs. registry

Rigid step-by-step pipelines scale every stage together and restart from scratch on mid-flow failure. The Task Registry lets each agent scale, upgrade, or resume independently.

Schema black box for non-IoT users

Column names and table relationships are opaque to analysts and managers. The Manager validates against the IoT catalog and never requires users to know SQL or schema details.

Cross-agent error recovery

Query failures can be data issues or code issues. Splitting retry logic between Planner and Executor avoids blind restarts and keeps conversation state intact.

### 📊 Deep Dive: Agentic Factory Analytics (EDAS)

**An event-driven, multi-agent analytics framework built with LangGraph that converts raw industrial PostgreSQL databases into interactive, conversational business intelligence.**

* **The Target Users:** Production managers, plant operators, and operations executives who require immediate, ad-hoc insights from live factory floor data.
* **The Core Bottleneck:** Traditional factory analytics relies on static dashboards (like Grafana or PowerBI). If an operator wants to know a complex correlation—such as *"Why did Power Tester 3 spike in energy consumption last Tuesday during the night shift?"*—they must wait for a data engineer to manually write, test, and run an SQL join query.
* **The Solution (EDAS):** A decoupled multi-agent coordinator where non-technical stakeholders ask analytical questions in plain English, and a team of specialized AI agents dynamically explore database structures, generate optimized queries, and deliver synthesized insights.

---

#### 🏗️ The Asynchronous Task Registry Architecture

Instead of using a brittle, sequential LLM chain where a single query failure breaks the entire run, EDAS is built using an **Asynchronous Task Registry decoupled state-machine** back-ended by PostgreSQL.

* **The Manager Agent:** Parses the user's natural language request, validates the intent, and initializes a parent job state within the registry.
* **The Planner Agent:** Polls the registry for pending jobs, reviews the database schema definitions, and maps out the chronological analytical steps needed to answer the question.
* **The Executor Agent:** Polls for planned steps, writes localized PostgreSQL syntax, runs the read-only execution block, and streams raw results back to the registry.

#### 🛠️ Core Engineering Challenges & Solutions

##### 1. Mitigating Schema Context Bloat (Handling Massive Tables)

* **The Challenge:** Feeding an entire enterprise relational database schema (dozens of tables, hundreds of columns, and data types) directly into an LLM context window causes extreme token bloat, high costs, and severe attention degradation—causing the planner to hallucinate invalid table joins.
* **The Solution:** Implemented a **Dynamic Schema Fetcher**. The database schema is stored natively inside a localized lookup table. Before the Planner Agent builds its roadmap, a lightweight semantic search routing step identifies *only* the specific tables and columns relevant to the user's query, injecting a minimized, hyper-focused schema definition into the prompt context window.

##### 2. Deterministic SQL Security Guardrails (Preventing Data Mutation)

* **The Challenge:** Running dynamically generated code against an active operational data store carries catastrophic risk. A minor agent hallucination or a malicious prompt injection could generate a `DROP TABLE`, `DELETE`, or `UPDATE` command, corrupting or deleting live manufacturing telemetry.
* **The Solution:** Built a two-layer security framework to enforce absolute non-destructive execution:
1. **Regex AST (Abstract Syntax Tree) Parser:** Every string generated by the Executor Agent passes through a strict inspection layer that blocks any query containing mutation keywords or non-`SELECT` statements.
2. **Transactional Rollbacks:** The query runs under a strictly restricted, read-only database user profile inside an isolated transaction block that auto-rolls back if any data modification is attempted.
